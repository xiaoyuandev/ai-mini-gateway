# 第三方接入说明：启动方式调整

## 文档目的

这份文档用于向接入 `ai-mini-gateway` 的第三方工具说明本次“启动使用方式”相关调整，并帮助判断这些调整是否会影响现有集成。

本文档只描述启动方式、默认参数和 CLI 行为调整，不涉及 HTTP API contract 变更。

## 本次调整内容

### 1. 支持安装后单命令启动

当前支持通过以下方式安装：

```bash
go install github.com/yuanjunliang/ai-mini-gateway@latest
```

安装完成后，用户可以直接执行：

```bash
ai-mini-gateway
```

这意味着第三方工具在“用户已安装 runtime”的前提下，可以直接用 `ai-mini-gateway` 拉起服务，而不再必须依赖：

```bash
go run ./cmd/gateway
```

或：

```bash
./bin/ai-mini-gateway
```

### 2. 启动参数支持默认值

当前以下参数已具备稳定默认值，用户不传参也可以直接启动：

1. `host=127.0.0.1`
2. `port=3457`
3. `data-dir=./data`
4. `models-cache-ttl=15s`

因此：

```bash
ai-mini-gateway
```

等价于：

```bash
ai-mini-gateway --host 127.0.0.1 --port 3457 --data-dir ./data --models-cache-ttl 15s
```

如果只需要覆盖个别参数，也可以只传必要项，例如：

```bash
ai-mini-gateway --port 8080
```

或：

```bash
ai-mini-gateway --data-dir /tmp/ai-mini-gateway
```

### 3. CLI 帮助行为已标准化

执行 `-h` 或 `--help` 时，程序现在只输出帮助信息并退出，不会再继续尝试启动服务。

这项调整主要影响通过 CLI 探测参数能力的工具链或安装脚本。

## 对第三方工具的影响评估

### 1. 对 HTTP API 接入没有影响

本次调整不涉及以下内容变更：

1. `/health`
2. `/capabilities`
3. `/v1/*`
4. `/admin/*`
5. `/runtime/status`

因此，已经按现有 HTTP contract 集成的第三方工具，不需要因为本次调整修改接口适配逻辑。

### 2. 对命令行拉起方式属于兼容性增强

本次调整是“增加更简单的启动入口”和“明确默认值”，不是破坏性修改。

以下已有方式仍然可用：

1. `go run ./cmd/gateway ...`
2. `./bin/ai-mini-gateway ...`
3. 使用 `--host`、`--port`、`--data-dir` 显式传参
4. 使用 `LOCAL_GATEWAY_RUNTIME_HOST`、`LOCAL_GATEWAY_RUNTIME_PORT`、`CORE_DATA_DIR` 注入环境变量

也就是说，现有第三方工具如果已经在显式传参或使用既有 binary 路径，可以继续保持原实现。

### 3. 可选地简化启动集成

如果第三方工具的目标是“尽量简化用户本地启动步骤”，现在可以在用户已安装后直接执行：

```bash
ai-mini-gateway
```

仅在存在明确运行要求时再覆盖参数，例如：

1. 指定固定端口
2. 指定固定数据目录
3. 指定固定监听地址

## 第三方工具需要特别注意的点

### 1. 默认 `data-dir` 与工作目录相关

当前默认值为：

```bash
./data
```

这表示数据目录是相对于“启动命令执行时的当前工作目录”解析的。

如果第三方工具希望数据目录稳定、可预测、与调用位置无关，建议继续显式传入：

```bash
--data-dir <absolute-or-controlled-path>
```

### 2. 默认端口仅适用于无冲突场景

当前默认端口为：

```bash
3457
```

如果第三方工具所在环境中可能存在端口占用，仍建议继续显式指定 `--port`，不要完全依赖默认值。

### 3. 现有显式传参逻辑无需回退

如果第三方工具已经实现了以下逻辑：

1. 明确设置 `--port`
2. 明确设置 `--data-dir`
3. 明确设置 `--host`
4. 通过环境变量注入运行参数

那么本次不要求修改，也没有必要为了“更短的命令”而主动改造现有稳定实现。

## 推荐给第三方工具的接入结论

### 建议一：已有稳定拉起逻辑的接入方

保持现状即可，无需因本次调整修改启动实现。

### 建议二：正在新增接入的工具

可以优先采用以下两类策略之一：

1. 最简策略：用户安装后直接执行 `ai-mini-gateway`
2. 稳定策略：执行 `ai-mini-gateway`，同时显式传入 `--port` 和 `--data-dir`

### 建议三：对运行环境要求较强的工具

如果你们对端口、数据目录、工作目录、进程隔离或多实例运行有要求，建议继续保留显式参数控制，不要依赖默认值。

## 结论

本次调整的核心是：

1. 新增安装后单命令启动能力
2. 明确默认启动参数
3. 标准化 CLI 帮助行为

对第三方工具而言，这些变更本质上是“兼容性增强”，不是协议破坏或运行时语义变更。

已经完成接入的工具通常不需要修改；希望简化接入体验的工具则可以利用新的默认启动方式减少启动参数和用户操作步骤。
